# MongoDB 设计

## 角色

MongoDB 是系统的主持久化存储，负责保存文章元数据和正文内容。对当前这个博客项目来说，只要 MongoDB 数据完整，就足以恢复公开内容模型。

后端通过 `MONGODB_URI` 和 `MONGODB_DATABASE` 连接 MongoDB；如果环境变量未设置，默认回退到 `mongodb://localhost:27017` 和数据库 `wanderlust`。

## 数据模型

当前后端只使用一个集合：

- `posts`：同时保存草稿和已发布文章

集合中的文档与后端 `Post` 结构体一一对应，核心字段包括：

- `id`：应用侧分配的整数 ID
- `slug`：文章公开路由使用的标识
- `title`、`summary`、`category`、`tags`、`author`
- `publishedAt`：发布时间字符串
- `readMinutes`：根据正文估算出的阅读时长
- `draft`：是否为草稿
- `featured`：是否为精选文章
- `accent`：前端展示使用的视觉强调色
- `bodyFormat`：`markdown` 或 `html`
- `body`：完整正文内容

这个模型的特点是简单直接，一篇文章的详情页、归档列表、管理端编辑器都可以由同一份文档派生出来，不需要额外拆表。

## 索引策略

后端启动时会先对 MongoDB 做连通性检测，然后确保 `slug` 上存在唯一索引。

当前索引设计只有一条：

- `slug` 唯一索引：保证公开文章 URL 不会冲突

目前没有给 `id` 增加唯一索引，也没有给列表排序、草稿过滤、精选查询单独建立二级索引。这在当前规模下是可接受的，因为该项目的目标是单站点、低写入频率的博客，而不是高并发 CMS。

## 读写流程

### 新建文章

创建文章时，后端会依次执行：

1. 规范化并校验请求体。
2. 查询当前最大的 `id`，分配下一个整数值。
3. 预留一个可用的 `slug`；如果冲突，则自动追加 `-2`、`-3` 等后缀。
4. 根据正文内容计算 `readMinutes`。
5. 将最终文档插入 `posts` 集合。

这意味着 `id` 不是数据库侧自增，而是应用侧生成。好处是实现轻量，代价是它更适合单实例或低写冲突场景。

### 文章列表

公开列表会通过 `draft != true` 过滤草稿，同时不返回 `body` 字段以减小传输体积。管理端列表则包含草稿。

两类列表都按以下规则排序：

- `publishedAt` 倒序
- `id` 倒序

### 单篇文章查询

公开文章详情按 `slug` 查询，并且同样排除草稿。管理端详情也按 `slug` 查询，但允许读取草稿内容。

### 精选文章约束

精选文章数量限制是在应用层完成的。当前后端会限制精选文章最多 3 篇，并禁止把草稿设置为精选。

## 部署设计

在 Docker Compose 中，MongoDB 以 `mongodb` 服务运行，镜像为 `mongo:7`。

当前容器设计包含：

- 命名卷 `mongodb-data` 挂载到 `/data/db`
- 基于 `mongosh ... db.runCommand({ ping: 1 })` 的健康检查
- 通过 `MONGODB_WIREDTIGER_CACHE_GB` 控制 `--wiredTigerCacheSizeGB`

这套参数明显偏保守，是为了适配低内存 VPS。部署环境默认把 WiredTiger cache 压到 `0.25GB`，以便在 `1GB` 主机上也能稳定运行。

## 备份与恢复

仓库已经包含 MongoDB 的备份恢复脚本：

- `scripts/backup-mongodb.sh`：导出目标数据库的 gzip archive，同时打包媒体文件
- `scripts/restore-mongodb.sh`：恢复数据库归档，并按需恢复媒体文件

默认备份目录在 `backups/mongodb/`，最近一次成功备份会同步到 `backups/latest-mongodb/`，方便运维取最近快照。

## 设计取舍

当前方案的优点：

- 数据模型面很小，维护成本低
- 单文档就能承载大多数读取场景，文章读取逻辑简单
- 通过 Docker 命名卷持久化，容器重建不会丢失正常业务数据

当前方案的边界：

- 还没有 migration 体系
- `id` 分配不适合高并发写入
- 数据库层面只有 `slug` 被强约束为唯一

对现在这个产品形态来说，这些取舍是合理的，因为它更像一个单编辑者博客系统，而不是多租户内容平台。

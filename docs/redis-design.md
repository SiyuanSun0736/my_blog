# Redis 设计

## 角色

Redis 不是系统主数据库，它当前只承担一个很明确的职责：

- 保存图片内容摘要，用于上传去重

即使 Redis 不可用，博客依然可以正常读文章、写文章；受影响的只有“重复图片秒返回已有路径”这一项优化能力。

## 为什么这里需要 Redis

管理端支持图片上传。如果没有摘要索引，同一张二进制图片被多次上传时，会反复写入媒体卷，产生重复文件。

Redis 在这里充当一个高速辅助索引，让后端可以快速回答一个问题：

- 这份二进制内容是否已经存过？

因此 Redis 是性能优化层和存储去重层，而不是系统事实来源。

## 键模型

后端使用的键前缀为：

- `wanderlust:media:digest:`

完整键形态为：

- `wanderlust:media:digest:{sha256}`

值的形态为：

- 公开媒体路径，例如 `/media/2026/05/<sha256>.png`

Redis 里不保存文章元数据，也不保存图片二进制本身，只保存“摘要 -> 已存在公开路径”的映射。

## 上传流程

管理端上传图片时，后端会执行以下步骤：

1. 通过 `BLOG_IMAGE_UPLOAD_MAX_BYTES` 限制请求体大小。
2. 校验文件类型是否属于允许的图片格式：gif、jpg、png、svg、webp。
3. 计算文件的 SHA-256 摘要。
4. 查询 Redis 中的 `wanderlust:media:digest:{sha256}`。
5. 如果键存在，且目标文件在磁盘上仍然存在，则直接返回已有路径，并标记本次命中缓存。
6. 如果键不存在，则把文件写入共享媒体目录，路径格式为 `YYYY/MM/<sha256>.<ext>`。
7. 将“摘要 -> 路径”的映射写回 Redis。

这套设计的核心是按内容去重，而不是按文件名去重。

## 降级策略

Redis 在运行时是可选组件。

只有配置了 `REDIS_ADDR` 时，后端才会创建 Redis 客户端；如果连通性检测失败，后端会记录日志，并退回到 no-op 的上传缓存实现。

这使系统具备两个关键特性：

- Redis 故障不会影响博客读路径
- Redis 故障不会阻断文章发布和图片上传本身

唯一的功能退化是：重复上传的图片不会被复用，而是会再次落盘。

## 持久化与内存策略

在 Docker Compose 中，Redis 以 `redis` 服务运行，镜像为 `redis:7-alpine`。

当前运行配置包括：

- 开启 AOF 持久化：`--appendonly yes`
- 开启快照规则：`--save 60 1000`
- 通过 `REDIS_MAXMEMORY` 控制最大内存
- 淘汰策略为 `allkeys-lru`
- 命名卷 `redis-data` 挂载到 `/data`

部署环境默认把 Redis 控制得比较小，当前仓库的部署默认值是 `64mb`；本地开发默认值更高，为 `256mb`。

## 与媒体清理的关系

Redis 的键与媒体文件通过摘要绑定。后端会定期扫描文章正文，找出仍被引用的 `/media/...` 文件，删除磁盘上长期未引用的媒体文件，并同步删除对应的 Redis 摘要键。

这样做的目的是避免 Redis 去重索引与真实磁盘文件逐渐偏离。

## 设计取舍

当前方案的优点：

- Redis 的职责单一，容易理解和维护
- 即使缓存全丢，系统事实来源仍然在 MongoDB 和媒体卷，恢复成本低
- 已经内置 no-op 降级实现，运行时退化比较平滑

当前方案的边界：

- Redis 没有用于会话、页面缓存或消息队列
- 摘要映射没有 TTL，依赖显式媒体清理来移除无效键
- 如果 Redis 被完全清空，旧媒体文件仍可访问，但去重能力要从后续上传逐步重新建立

对当前项目来说，这个取舍是合理的，因为 Redis 的定位就是减少重复媒体存储，而不是保存不可替代的业务状态。
